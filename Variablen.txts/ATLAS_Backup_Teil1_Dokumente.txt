#### Deliverables

- Repo mit Next.js + Prisma, `prisma/schema.prisma` mit MVP-Modellen, Migrationen angewendet.
- README mit Anleitung: `npm install`, `.env` (DATABASE_URL), `npx prisma migrate dev`.

### Woche 2: Seed-Daten

**Ziel:** Seed-Skript mit Kategorien, Tools, Integrationen und Stacks. Daten in DB laden und prüfen.

#### Tasks (abzuarbeiten)

- [ ] **T2.1** Seed-Skript anlegen: mind. 20–50 Kategorien, 100+ Tools (name, slug, description, websiteUrl, pricing), Category-Zuordnung.
- [ ] **T2.2** Seed um Integrationen erweitern: `ToolIntegration`-Einträge (toolId, integratesWithId) — Ziel mind. 100+ Kanten für erste 100 Tools.

- [ ] **T2.4** Seed ausführen, Daten in DB prüfen (Tools, Integrationen zählbar). README um `npx prisma db seed` ergänzen.

- `prisma/seed.ts` (oder äquivalent) mit Tools, Kategorien, Integrationen, Stacks.
- Lauffähige DB mit geseedeten Daten.

#### Akzeptanzkriterien

- `npx prisma db seed` läuft fehlerfrei.

### Woche 3: AI Stack Generator (Kernprodukt MVP)

**Ziel:** Nutzer gibt ein Problem ein (z. B. „I want to automate YouTube videos") → Atlas liefert einen konkreten AI-Stack (Schritte + Tools). Stack kann gespeichert werden und erhält eine öffentliche Seite.

#### Tasks (abzuarbeiten)

- [ ] **T3.1** API: `POST /api/v1/generate-stack` — Input: Text-Prompt. Output: Steps (geordnet) mit Tool-Referenzen aus der DB. Siehe [API-MVP](./spec.md#API-MVP).
- [ ] **T3.2** API: `POST /api/v1/stacks` — Speichern eines generierten Stacks: Titel (aus Prompt oder generiert), Slug, Steps (toolId, order). Legt `Stack` + `StackStep`-Einträge an. Gibt Slug/URL zurück.
- [ ] **T3.3** Seite `/generate`: Prompt eingeben → Stack generieren → Preview anzeigen → "Save & Get Link" speichert und leitet zu `/stacks/[slug]` weiter.
- [ ] **T3.4** Generierte Stack-Seite: `/stacks/[slug]` muss auch für neu erzeugte Stacks funktionieren (dynamisch aus DB). Bereits in Woche 4 vorgesehen; sicherstellen, dass Stack-Generator Stacks in derselben Struktur speichert wie Seed-Stacks.

#### Deliverables

- `app/generate/page.tsx`, Komponenten: GenerateStackForm, GeneratedStackPreview.
- API: `app/api/v1/generate-stack/route.ts`, `app/api/v1/stacks/route.ts` (POST).
- LLM-Integration (env: OPENAI_API_KEY oder ähnlich), Prompt-Template das Tools aus DB nutzt.

#### Akzeptanzkriterien

- Nutzer kann auf `/generate` einen Satz eingeben und erhält einen sinnvollen Tool-Workflow.
- „Save & Get Link" speichert den Stack und leitet auf `/stacks/[slug]` weiter.
- Seite `/stacks/[slug]` zeigt den neu erzeugten Stack korrekt an.

#### Relevante Sektionen

- [spec.md: API-MVP](./spec.md#API-MVP), [spec.md: ACCEPTANCE-CRITERIA](./spec.md#ACCEPTANCE-CRITERIA)

### Nach dem MVP (nicht in dieser Checkliste)

- Clone-Button mit echtem Auth und „Clone to workspace".
- Galaxy Map 3D (Three.js, Section 41).
- Erweiterter AI Copilot, manueller Stack Builder UI.
- Reviews, Newsletter, Recommendation Engine.

### Proof Hypothesis — ab Launch messen

- **Stacks generiert** (über `/generate`).
- **Clone-Button-Klicks** (auch wenn Stub).
- **Traffic:** `/generate`, `/stacks` vs. `/tools`.

**Hypothese:** Nutzer sagen "I generated an AI stack with Atlas" — nicht "I browsed AI tools".

### Kurzreferenz: Was NICHT bauen (MVP)

- Kein Auth (Supabase/Login) — Phase 2.
- Keine User-Tabellen, keine Saved Tools, kein Dashboard.
- Keine Reviews, keine Ratings, kein Newsletter.
- Keine 3D Galaxy Map, kein manueller Stack Builder. Keine Zahlungen (Stripe) — Phase 2.

# ATLAS MVP Spec (Single Source of Truth)

NAMING-POLICY:
- AIAstralis = Firma
- ATLAS = Produkt
- Dateinamen/Ordner: kebab-case, keine Leerzeichen

LANGUAGE-POLICY:
- Execution/Engineering Artefakte: Deutsch
- Brand/Marketing Text (optional): Englisch

REQ-MVP-SCOPE — Nur diese Features sind MVP:
1) AI Stack Generator
2) Stack Pages
3) Tool Relationship Graph (nur Datenerfassung)

Explizit nicht im MVP: Auth, Payments, Reviews, Newsletter, Recommendation Engine, 3D/Three.js.

REQ-MVP-ROUTES — Nur diese Routen bauen:
/ | /generate | /stacks | /stacks/[slug] | /tools | /tools/[slug] | /categories | /search

DB-MVP-SCHEMA — Minimum-Modelle:
Tool | Category | ToolCategory (Join) | ToolIntegration (Edges) | Stack | StackStep | Tag | ToolTag (Join)

Invarianten:
- slug ist unique pro Entity-Typ
- StackStep hat eindeutige Reihenfolge pro Stack
- ToolIntegration darf keine Self-Edges enthalten

API-MVP:
POST /api/v1/generate-stack — Input: Freitext-Prompt. Output: geordnete Steps mit Tool-Referenzen aus DB.
POST /api/v1/stacks — Input: Titel/Slug + Steps. Output: slug + öffentliche URL.
GET /api/v1/stacks | GET /api/v1/stacks/[slug] | GET /api/v1/tools | GET /api/v1/tools/[slug] | GET /api/v1/categories

RELATIONSHIP-GRAPH:
Beim Speichern eines Stacks werden Tool-Kombinationen erfasst. Keine UI-Graph-Visualisierung im MVP.

ACCEPTANCE-CRITERIA:
- /generate erzeugt sinnvollen Workflow aus DB-Tools
- Save & Get Link speichert Stack, öffnet /stacks/[slug]
- /stacks listet Seed + generierte Stacks
- /tools und /tools/[slug] funktionieren inkl. Suche/Filter
- Relationship-Tracking läuft bei gespeicherten Stacks

ATLAS — Quality Gates (Hard Mode) Developer-Checkliste

MVP-Pflicht: Datenmodell-Constraints + grundlegende Rate-Limits.

Harte Indexing-Gates:
- >= 2 data-rich Sections aus DB
- >= 1 unique Narrative
- >= 3 kontextuelle interne Links
- Unique title, meta description, FAQ

Rate Limits:
- GET /api/v1/tools: 60/min
- GET /api/v1/tools/search?q=: 30/min
- POST /api/v1/tools/submit: 5/hour
- POST /api/v1/reviews: 10/day
- GET /api/v1/og-image: 30/min

Postgres DDL (MVP-Kern):
tools, categories, tool_categories, tool_integrations, stacks, stack_steps

Key Constraints:
- tools.slug, categories.slug, stacks.slug: unique
- stack_steps: unique (stack_id, step_order)
- tool_integrations: no self-edges (tool_id <> integrates_with_id)
- stacks.difficulty: check IN ('easy','medium','hard')

SEO Gates (seo_pages table):
indexing_state: enum('indexable','noindex','blocked')
quality_passed: boolean
data_rich_sections_count >= 2
unique_narrative_present: true
internal_links_count >= 3
unique_meta_present: true

Rate Limit Key-Schema:
rl:public:ip:{ip} → 60/1m
rl:search:ip:{ip} → 30/1m
rl:submit:user:{userId} → 5/1h
rl:review:user:{userId} → 10/1d
rl:og:ip:{ip} → 30/1m

ATLAS OS — Agent Architecture

Agent pipeline:
User Goal → Architect Agent → Task Planner Agent → Research Agent → Builder Agent → Deployment Agent

Core Agents:
1. Architect Agent — system design, Input: user goal, Output: system architecture
2. Task Planner Agent — breaks architecture into executable tasks, Output: task graph
3. Research Agent — discovers tools from ProductHunt, GitHub, HackerNews, Reddit, Futurepedia
4. Builder Agent — generates stack pages, SEO pages, API integrations, workflows
5. Deployment Agent — deploys pages, updates databases, triggers distribution

Agent Runtime Components:
Agent Registry | Task Queue | Scheduler | Executor | Memory

Agent Registry functions: register_agent() | get_agent() | list_agents()

Task Queue example:
{ "id":"task_1", "agent":"research", "description":"discover AI tools", "dependencies":[] }

Scheduler logic: task ready → dependencies satisfied → agent available → execute

Agent Memory — Stored data: task results, tool database, workflow graph, agent outputs
Memory types: short-term | long-term

Discovery Agents: producthunt_scraper | github_scraper | reddit_scraper → tools.json

Content Generation Agents: Tool pages, SEO pages, Ranking pages
Example URLs: /tools/chatgpt | /ai-tools-for-marketing | /best-ai-writing-tools

Distribution Agents: Twitter/X | Reddit | Newsletter | Product Hunt

Scaling Strategy:
Phase 1: Manual discovery, Basic agents, Static generation
Phase 2: Autonomous discovery, Content generation, Distribution agents
Phase 3: Fully autonomous ATLAS OS

Long-Term Vision: ATLAS = AI operating system for building AI systems

app = FastAPI(title="AIAtlas API")
class Tool(BaseModel):
    name: str
    description: str
TOOLS = []
@app.get("/") def root(): return {"message":"AIAtlas API running"}
@app.post("/tools") def add_tool(tool: Tool): TOOLS.append(tool); return {"status":"added","tool":tool}
@app.get("/tools") def list_tools(): return TOOLS
