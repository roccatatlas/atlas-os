1. Technologie-Stack


1.1 Frontend
Framework: Next.js (App Router, SSR + SSG)
Sprache: TypeScript (tsconfig.json, strict mode)
Styling: Tailwind CSS + PostCSS
Komponenten: Eigenes Design-System (Navbar, Footer, Layout, ui.tsx mit 20+ Komponenten)
Charts: VisualEngine / VisualEngineV2 (eigene Chart-Abstraktion)
State: React Context (SearchContext)

1.2 Backend
Framework: Express.js + TypeScript (apps/api/src/main.ts)
45+ REST-Endpunkte: /search, /tools, /trending, /benchmark, /compare, /models, /workflows, /agents, /insights, /changes, /market-map …

4. Service-Layer & Algorithmen


4.1 Ranking Engine (services/ranking-engine/ranking.ts)
Gewichteter 5-Signal-Score. Standardgewichte (konfigurierbar):

Signal
Gewicht
Berechnung
traffic
25%
Besuche letzte 30 Tage, normalisiert
reviews
30%
Normalisierter Count × (avg_rating / 5)
features
15%
Anzahl Features normalisiert
relevance
20%
1 − vector_distance (aus Suche)
recency
10%
Exponentieller Decay e^(−days/130), Halbwertszeit 90 Tage


4.2 Vector Search (services/vector-search/search.ts)
Index: HNSW cosine_ops (pgvector) für approximate nearest neighbor
Dimensionen: 1536 (text-embedding-3-small)
Max Distance: 0.8 (konfigurierbar)
Funktionen: vectorSearch(vector, options), findSimilarTools(toolId), upsertEmbedding(toolId, embedding)
Kategorie-Filter: Optional als SQL-AND-Bedingung, nicht Post-Filter → performant

4.3 Recommendation Engine (hybrid)
Quellen: Vector Search (ANN) + Knowledge Graph (graph_edges) kombiniert
Deduplication: Set<number> über tool_ids vor Ranking
Graph-Typen: similar_to + alternative_to Relations werden parallel abgefragt
Ranking: Gemeinsames rankTools() mit relevance-Boost (0.3) für Graph-Matches

4.4 Trust Score (services/trust-engine/trust.ts)
5-dimensionaler Composite-Score 0-100:
review_score (25%): (avg_rating/5 × 80) + min(review_count/10, 20)
company_score (20%): log10(funding) × 5 + Unternehmensalter × 5 + 30 wenn verified
transparency_score (20%): open_source +50, api_available +30, github_url +20
longevity_score (15%): Tage seit Launch / 3.65 — 100 bei ~1 Jahr
community_score (20%): (stars/1000 × 50) + (monthly_traffic/100000 × 50)

4.5 Model Router (services/model-router/router.ts)
Input: task_type, priority (quality/speed/cost), max_latency_ms, context_needed
Output: RoutingDecision mit recommended_model, reason, estimated_cost, estimated_latency_ms
Endpunkt: POST /models/route

4.6 Trend Engine (services/trend-engine/trends.ts)
Vergleicht aktuellen Zeitraum mit vorherigem gleicher Länge:
rising: growth_pct > 10%
stable: growth_pct −10% bis +10%
declining: growth_pct < −10%
Heatmap: Wöchentliches Kategorie-Wachstum via CTE, Endpunkt: GET /trends/heatmap

9. Wichtige Architektur-Muster


9.1 Materialisierte Rankings (Caching-Pattern)
Statt Rankings live zu berechnen, schreibt der Worker stündlich in tool_rankings (rank, score, calculated_at). API liest nur noch die Tabelle ab. Redis-Invalidierung bei Änderung:
Schreiben: DELETE FROM tool_rankings → INSERT alle Ränge (Worker, stündlich)
Lesen: SELECT * FROM tool_rankings ORDER BY rank — O(1)
Invalidierung: invalidate('search:*') in Redis nach jedem Ranking-Run

9.2 Hybrid Vector + Graph Search
Die Recommendation Engine kombiniert zwei Quellen parallel:
1. pgvector ANN (cosine distance < 0.8) für semantische Ähnlichkeit
2. graph_edges (similar_to + alternative_to) für explizite Relationen
Merge via Set<number> zur Deduplizierung, dann einheitliches rankTools()

9.3 Tiered Worker Architecture
Jobs nach Kritikalität gestaffelt:
Tier 1 (10s): Analytics-Flush — kritisch, minimale Latenz
Tier 2 (30m-1h): Enrichment, Performance-Monitor, Rankings
Tier 3 (2-6h): Graph-Relations, Similarity, Trust-Scores
Tier 4 (6-24h): Discovery, Change-Tracker, Insights, SEO, Content, Screenshots
Tier 5 (wöchentlich): Newsletter, Workflow-Seeds

9.4 Auto-generierter Content-Stack
ai_reviews: AI-generierte Reviews mit strengths, weaknesses, best_for — UNIQUE(tool_id, generated_by)
ai_tutorials: Tutorials mit Difficulty-Level, published-Flag
ai_comparisons: X-vs-Y Vergleichstexte — UNIQUE(tool_a_id, tool_b_id)
auto_content: Generischer Content-Store mit Status-Workflow: draft → review → published → archived
quality_score: Jedes Content-Stück bekommt einen Qualitäts-Score vor der Publikation

9.5 Multi-Source Ingestion Pattern
ingestion_sources definiert alle Quellen (mit config JSONB für Quell-spezifische Parameter)
ingestion_jobs protokolliert jeden Lauf (tools_found, tools_new, started_at, finished_at, error)
ingestion_logs für vollständiges Job-Logging (level: info/warn/error, metadata JSONB)
Deduplication: UNIQUE(source, external_id) auf tools-Tabelle — Upsert statt Insert

9.6 API-Key-Tiers
Dreistufiges Zugangsmodell:
free: 10.000 Requests/Monat
pro: Höheres Limit, erweiterte Endpunkte
enterprise: Unlimitiert, dedizierter Support
Tracking in api_usage_log (endpoint, status_code, latency_ms) für detaillierte Auswertung
