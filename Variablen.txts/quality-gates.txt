## ATLAS — Quality Gates (Hard Mode) Developer-Checkliste

Diese Checkliste ist so geschrieben, dass ein Entwickler sie 1:1 implementieren kann: DB-DDL (Postgres), Constraints, Rate-Limit-Key-Schema, Audit-Log-Schema und die konkreten Gates aus dem Blueprint.

**Single Source of Truth (MVP):** [spec.md](../mvp/spec.md)

MVP-Pflicht: Datenmodell-Constraints + grundlegende Rate-Limits. Alles, was darüber hinausgeht (Programmatic SEO Landings, Compare/Alternatives), ist Post-MVP, außer es wird explizit in [REQ-MVP-SCOPE](../mvp/spec.md#REQ-MVP-SCOPE) aufgenommen.

### Geltende Seitentypen (aus Blueprint)

Routes/URL-Patterns:
- `/tools/{slug}`
- `/stacks/{slug}`
- `/category/{slug}`
- `/compare/{slugA}-vs-{slugB}` oder `/compare/{slugA}/{slugB}`
- `/alternatives/{slug}`
- Programmatic SEO Landings (je nach Strategie): `/ai-tools-for/{use_case}`, `/best-ai-tools-for/{use_case}`, `/use-cases/{industry}/{goal}`

Hinweis: Für das MVP sind die verbindlichen Routen in [REQ-MVP-ROUTES](../mvp/spec.md#REQ-MVP-ROUTES) definiert.

### Harte Indexing-Gates (aus Blueprint)

Eine programmatic Seite ist nur indexierbar, wenn alle erfüllt sind:
- ≥ 2 data-rich Sections (Tabellen/Graphen) aus DB
- ≥ 1 unique Narrative (nicht Boilerplate)
- ≥ 3 kontextuelle interne Links
- Unique `title`, `meta description`, `FAQ` (keine templated Duplikate)

Wenn ein Gate failt:
- Seite ausliefern, aber `noindex,follow`
- Human Review Queue, falls Top-Traffic

Canonical/Redirect/Params:
- Genau 1 Canonical pro URL und Seitentyp
- Query-Params erzeugen keine neuen Canonicals
- Pagination: canonical auf paginierte URL + `rel="prev/next"`
- Slug-Änderungen immer als 301 + Slug-History, alte Slugs nie wiederverwenden

### Rate Limits (aus Blueprint)

Public API Rate Limits (Auszug):
- `GET /api/v1/tools` 60/min
- `GET /api/v1/tools/search?q=` 30/min
- `POST /api/v1/tools/submit` 5/hour
- `POST /api/v1/reviews` 10/day
- `GET /api/v1/og-image` 30/min

## 1) Datenmodell: MVP-Kern (Tools, Categories, Stacks)

Das ist der Minimum-Unterbau für `/tools`, `/tool/[slug]`, `/stacks`, `/stacks/[slug]`, `/categories`, `/search`.

### 1.1 Postgres DDL (MVP)

```sql
create table if not exists tools (
  id uuid primary key default gen_random_uuid(),
  slug text not null unique,
  name text not null,
  description text,
  website_url text,
  pricing text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists categories (
  id uuid primary key default gen_random_uuid(),
  slug text not null unique,
  name text not null,
  description text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists tool_categories (
  tool_id uuid not null references tools(id) on delete cascade,
  category_id uuid not null references categories(id) on delete cascade,
  primary key (tool_id, category_id)
);

create table if not exists tool_integrations (
  tool_id uuid not null references tools(id) on delete cascade,
  integrates_with_id uuid not null references tools(id) on delete cascade,
  created_at timestamptz not null default now(),
  primary key (tool_id, integrates_with_id),
  constraint tool_integrations_no_self check (tool_id <> integrates_with_id)
);

create table if not exists stacks (
  id uuid primary key default gen_random_uuid(),
  slug text not null unique,
  title text not null,
  description text,
  is_public boolean not null default false,
  views int not null default 0,
  clones int not null default 0,
  difficulty text not null default 'medium',
  cost_monthly numeric,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint stacks_difficulty check (difficulty in ('easy','medium','hard'))
);

create table if not exists stack_steps (
  id uuid primary key default gen_random_uuid(),
  stack_id uuid not null references stacks(id) on delete cascade,
  tool_id uuid not null references tools(id) on delete restrict,
  step_order int not null,
  role text not null,
  notes text,
  constraint stack_steps_order_nonnegative check (step_order >= 0),
  unique (stack_id, step_order)
);

create index if not exists stack_steps_stack_id_idx on stack_steps(stack_id);
create index if not exists tool_integrations_tool_id_idx on tool_integrations(tool_id);
create index if not exists tool_integrations_integrates_with_id_idx on tool_integrations(integrates_with_id);
```

### 1.2 Constraints, die im MVP sofort helfen

- `tools.slug`, `categories.slug`, `stacks.slug` unique
- `stack_steps` unique `(stack_id, step_order)`
- `tool_integrations` verhindert Self-Edges

## 2) Datenmodell: SEO-Quality Gates (Indexing als DB-State)

Ziel: Indexing ist kein Default. Jede indexierbare URL bekommt einen DB-State, den die App nutzt, um Canonical/Robots/Meta zu steuern.

### 2.1 Postgres DDL (SEO Gates)

```sql
create type indexing_state as enum ('indexable', 'noindex', 'blocked');
create type page_type as enum ('tool', 'stack', 'category', 'compare', 'alternatives', 'landing');

create table if not exists seo_pages (
  id uuid primary key default gen_random_uuid(),
  page_type page_type not null,
  locale text not null default 'en',
  canonical_url text not null,
  path text not null,
  source_entity_id uuid,
  source_hash text not null,
  fingerprint text,
  indexing_state indexing_state not null default 'noindex',
  quality_version int not null default 1,
  quality_passed boolean not null default false,
  data_rich_sections_count int not null default 0,
  unique_narrative_present boolean not null default false,
  internal_links_count int not null default 0,
  unique_meta_present boolean not null default false,
  last_checked_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint seo_pages_counts_nonnegative check (
    data_rich_sections_count >= 0 and internal_links_count >= 0
  ),
  constraint seo_pages_unique_per_locale_path unique (locale, path),
  constraint seo_pages_unique_canonical unique (canonical_url)
);

create index if not exists seo_pages_source_entity_idx on seo_pages(source_entity_id);
create index if not exists seo_pages_fingerprint_idx on seo_pages(fingerprint);
create index if not exists seo_pages_indexing_state_idx on seo_pages(indexing_state);
```

### 2.2 Gate-Log (debuggbar + auditierbar)

```sql
create table if not exists seo_page_quality_results (
  id uuid primary key default gen_random_uuid(),
  seo_page_id uuid not null references seo_pages(id) on delete cascade,
  run_id uuid not null,
  gate text not null,
  passed boolean not null,
  details jsonb,
  created_at timestamptz not null default now(),
  unique (seo_page_id, run_id, gate)
);

create index if not exists seo_page_quality_results_page_idx on seo_page_quality_results(seo_page_id);
```

### 2.3 Gate-Definition als Checkliste (Code)

Für jede generierte Seite (oder beim Publish) muss der Worker diese Felder setzen:
- `data_rich_sections_count` (≥ 2)
- `unique_narrative_present` (true)
- `internal_links_count` (≥ 3)
- `unique_meta_present` (true)

Dann:
- wenn alle erfüllt: `quality_passed=true`, `indexing_state='indexable'`
- sonst: `quality_passed=false`, `indexing_state='noindex'`

Meta/Robots-Regeln:
- `indexing_state='noindex'` → `robots: noindex,follow`
- `indexing_state='blocked'` → 404 oder `robots: noindex,nofollow` (nur falls wirklich nötig)

## 3) Slug-History + 301 Redirects (SEO Safety)

Der Blueprint enthält bereits Prisma-Modelle `ToolSlugHistory` und `StackSlugHistory`. Das Minimum für die App ist:
- Bei jedem Rename: Insert in Slug-History
- Bei Request auf altem Slug: 301 auf neuen Slug
- Alte Slugs nie wiederverwenden

### 3.1 Postgres DDL (Slug History, generisch)

```sql
create type slug_entity_type as enum ('tool', 'stack');

create table if not exists slug_history (
  id uuid primary key default gen_random_uuid(),
  entity_type slug_entity_type not null,
  entity_id uuid not null,
  locale text not null default 'en',
  old_slug text not null,
  new_slug text not null,
  created_at timestamptz not null default now(),
  unique (entity_type, locale, old_slug)
);

create index if not exists slug_history_entity_idx on slug_history(entity_type, entity_id);
```

## 4) Abuse/Moderation + Audit Log (append-only)

Das ist der Unterbau für Submissions/Reviews/Stack-Approval und ist im Blueprint als Non-Negotiable beschrieben.

### 4.1 Postgres DDL (Moderation Status + Trust)

```sql
create type tool_submission_status as enum ('pending', 'in_review', 'approved', 'rejected');
create type tool_review_status as enum ('pending', 'approved', 'rejected', 'flagged');
create type stack_status as enum ('draft', 'pending', 'approved', 'rejected');

create table if not exists tool_submissions (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  url text not null,
  description text,
  category_slug text,
  submitter_email text,
  status tool_submission_status not null default 'pending',
  admin_note text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists tool_reviews (
  id uuid primary key default gen_random_uuid(),
  tool_id uuid not null references tools(id) on delete cascade,
  user_id uuid,
  rating int not null,
  review_text text,
  status tool_review_status not null default 'pending',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint tool_reviews_rating_range check (rating between 1 and 5)
);

alter table stacks
  add column if not exists status stack_status not null default 'draft';
```

### 4.2 Postgres DDL (Audit Log, append-only)

```sql
create type audit_target_type as enum ('tool_submission', 'review', 'stack', 'user');
create type audit_action as enum ('approve', 'reject', 'flag', 'shadow_ban', 'restore');

create table if not exists audit_log (
  id uuid primary key default gen_random_uuid(),
  actor_id uuid,
  target_type audit_target_type not null,
  target_id uuid not null,
  action audit_action not null,
  reason text,
  meta jsonb,
  created_at timestamptz not null default now()
);

create index if not exists audit_log_target_idx on audit_log(target_type, target_id, created_at);
create index if not exists audit_log_actor_idx on audit_log(actor_id, created_at);
```

Append-only Enforcement (DB):
- Keine Updates/Deletes auf `audit_log` granten
- Optional Trigger, der Updates/Deletes blockt (falls ihr super-hart sein wollt)

## 5) Rate Limiting: Key-Schema (Upstash/Redis)

Ziel: gleiche Limits wie in der API-Tabelle, aber mit sauberen Keys pro Use-Case.

### 5.1 Key-Schema

Keys:
- Public Read: `rl:public:ip:{ip}`
- Search: `rl:search:ip:{ip}`
- Tool Submit:
  - wenn `userId` vorhanden: `rl:submit:user:{userId}`
  - sonst: `rl:submit:ip:{ip}`
- Reviews:
  - `rl:review:user:{userId}`
- OG Image:
  - `rl:og:ip:{ip}`

### 5.2 Windows/Thresholds (Blueprint-konform)

- `rl:public:*` → 60 / 1m
- `rl:search:*` → 30 / 1m
- `rl:submit:*` → 5 / 1h
- `rl:review:*` → 10 / 1d
- `rl:og:*` → 30 / 1m

## 6) Implementierungs-Reihenfolge (DoD)

### 6.1 MVP (sofort)

- DB: `tools`, `categories`, `tool_categories`, `tool_integrations`, `stacks`, `stack_steps`
- SEO: `seo_pages`, `seo_page_quality_results`
- Runtime:
  - Meta pro Route setzt canonical
  - Robots tag folgt `seo_pages.indexing_state`
  - Sitemap nimmt nur `indexing_state='indexable'`
- Redirect:
  - Slug History Insert bei Rename + 301 Lookup

### 6.2 Post-MVP (wenn Submissions/Reviews live gehen)

- DB: `tool_submissions`, `tool_reviews`, `audit_log`, optional `slug_history` (oder Prisma-Varianten)
- API:
  - Submission Pipeline: duplicate check, status workflow
  - Review Moderation: auto-approve wenn 3+ approved reviews + keine flagged history
  - Jede Mod-Aktion schreibt `audit_log`
